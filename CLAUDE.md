# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`fox` (crate name `ferrumox`) is a high-performance local LLM inference server in Rust — a drop-in replacement for Ollama. It wraps `llama.cpp` (vendored as a Git submodule) and serves GGUF models over both **OpenAI-compatible** (`/v1/*`) and **Ollama-compatible** (`/api/*`) HTTP APIs. Default port is `8080`.

The binary is `fox`; a second binary `fox-bench` (`src/bin/bench.rs`) benchmarks a running server against Ollama.

## Build, test, lint

```bash
# Clone REQUIRES submodules (llama.cpp lives in vendor/llama.cpp)
git clone --recurse-submodules ...   # or: git submodule update --init --recursive

cargo build --release                # GPU backend auto-detected at build time
FOX_SKIP_LLAMA=1 cargo build         # stub build — no llama.cpp/GPU drivers needed

cargo check                          # fast type-check
cargo fmt                            # required before commit; CI rejects unformatted
cargo clippy --all-targets --features test-helpers -- -D warnings
```

**`FOX_SKIP_LLAMA=1` is the key escape hatch.** `build.rs` compiles llama.cpp via CMake (slow, needs the submodule + a C/C++ toolchain). Setting `FOX_SKIP_LLAMA=1` sets `cfg(fox_stub)` and swaps in a stub model so the crate builds and tests run without GPU drivers or the submodule. Use it for any work that doesn't touch real inference. CI runs entirely with it set.

### Running CI checks locally

`make ci` runs exactly what CI runs — do this before pushing:

```bash
FOX_SKIP_LLAMA=1 cargo fmt --all -- --check
FOX_SKIP_LLAMA=1 cargo clippy --all-targets --features test-helpers -- -D warnings
FOX_SKIP_LLAMA=1 cargo test --all --features test-helpers
```

`make setup` installs a pre-push git hook that runs these automatically.

### Tests

```bash
cargo test --lib                                    # unit tests in src/ (no server needed)
FOX_SKIP_LLAMA=1 cargo test --all --features test-helpers   # full suite as CI runs it
cargo test --test integration -- --test-threads=1  # tests/integration.rs (needs a running server)
cargo test <name>                                   # single test by name substring
```

The `test-helpers` feature gate exposes `StubModel`, `EngineEntry::for_test`, `ModelRegistry::preload_for_test`, and `src/api/test_helpers.rs` for integration tests in `tests/`. Tests that exercise the HTTP layer need this feature.

### Run the server during development

```bash
make download-model      # fetch a default GGUF into models/ (uses Docker + huggingface_hub)
make run                 # build + serve with models/ model
make dev                 # same with RUST_LOG=debug
# or directly:
./target/release/fox serve --model-path <path.gguf> --port 8080
```

## Architecture

Request flow, HTTP → token stream:

```
CLI (src/cli/)  ──fox serve──▶  AppState { ModelRegistry + config }  ──▶  Axum server
                ──fox run────▶  load one model, run inference loop directly

HTTP request → src/api/router.rs (router + AppState)
  → handler in src/api/v1/ (OpenAI) or src/api/ollama/ (Ollama)
  → registry.get_or_load(model) → Arc<InferenceEngine>   (loads on first request, "lazy loading")
  → handler formats prompt, builds InferenceRequest, submits to engine
  → engine run loop allocates KV blocks via Scheduler, decodes tokens
  → tokens stream back over an mpsc channel
  → handler emits SSE (OpenAI) or NDJSON (Ollama)
  → client disconnect makes send() fail → engine preempts the request and frees GPU memory
```

Module map (note: this layout supersedes the older flat paths described in `CONTRIBUTING.md`):

- **`src/api/`** — HTTP layer.
  - `router.rs` builds the Axum router + `AppState`; `routes.rs` lists routes.
  - `v1/` = OpenAI-compatible handlers (`chat`, `completions`, `embeddings`, `models`).
  - `ollama/` = Ollama-compatible handlers (`chat`, `generate`, `embed`, `management`).
  - `shared/` = inference + streaming helpers reused by both API families (`inference.rs`, `streaming.rs`, `digest.rs`, `extractor.rs`).
  - `types/` = request/response types split by surface (`v1`, `ollama`, `embeddings`, `tools`, `pull`, `shared`).
  - `auth.rs` = optional `FOX_API_KEY` bearer-token middleware; `pull_handler.rs` = `POST /api/pull` SSE download from HuggingFace.
- **`src/model_registry/`** — `ModelRegistry` = `DashMap<String, EngineEntry>` with LRU eviction (`--max-models`, `--keep-alive-secs`). `get_or_load()` loads a model on first request and starts its engine loop; `EngineEntry` holds `Arc<InferenceEngine>` and aborts the loop on `Drop` (eviction). `loader.rs` resolves model names/aliases to GGUF files.
- **`src/engine/`** — `InferenceEngine` receives `InferenceRequest`s and runs the decode loop. `model/llama_cpp/` wraps the llama.cpp C bindings (FFI in `engine/ffi.rs`, generated by `bindgen` in `build.rs`); `model/stub.rs` + `model/llama_cpp/stub.rs` provide the `fox_stub` no-op model. `sampling.rs`, `logits.rs`, `output_filter.rs` handle token selection and filtering.
- **`src/scheduler/`** — continuous batching + prefix caching. `Scheduler` is a priority queue assigning KV blocks to requests; `prefix_cache.rs` reuses already-processed shared prefixes (the core "conversations get faster over time" optimization); `batch.rs` / `schedule.rs` implement continuous batching so concurrent requests are processed together, not serialized.
- **`src/kv_cache/`** — `KVCacheManager`, a PagedAttention-style ref-counted block manager (`--block-size`, `--swap-fraction`) that prevents fragmentation under concurrent load and supports KV quantization (`f16`/`q8_0`/`q4_0`).
- **`src/cli/`** — subcommands: `serve`, `run`, `bench`, `bench-kv`, `pull`, `list`, `rm`, `show`, `ps`, `models`, `search`, `alias`. Each is `src/cli/<cmd>.rs` exposing `<Cmd>Args` (clap `Parser`) + `run_<cmd>()`; dispatched from the `Command` enum in `src/cli/mod.rs`.
- **`src/config.rs`** — loads `~/.config/ferrumox/config.toml`. Every CLI flag also has a `FOX_*` env var and a config-file key (precedence: flag > env > config file). Aliases live in `~/.config/ferrumox/aliases.toml`.
- **`registry.json`** (repo root) — built-in model catalog mapping short aliases → HuggingFace repo + GGUF filename pattern, consumed by `fox pull`/`fox models`.

### GPU backend selection (`build.rs`)

`build.rs` builds llama.cpp with `GGML_BACKEND_DL=ON`, so GPU backends are compiled as dynamically-loaded shared libs and **the actual backend is chosen at runtime** — one binary runs on CPU/CUDA/ROCm/Vulkan/Metal. At build time it auto-enables backends based on what's on the host: CUDA (`nvcc`/`CUDACXX`), Metal (macOS), ROCm/HIP (`hipcc`/`HIPCC`), Vulkan (`glslc`/`VULKAN_SDK`/`vulkan.h`). Do **not** explicitly link `ggml-cpu`/`ggml-cuda`/`ggml-metal` — they are dlopened via the backend registry.

## Conventions when extending

- **New API endpoint**: add types under `src/api/types/`, write the handler in the matching `v1/` or `ollama/` module (reuse `shared/` helpers), register the route in `src/api/router.rs`/`routes.rs`, document under `docs/api/`.
- **New CLI command**: add `src/cli/<cmd>.rs` with `<Cmd>Args` + `run_<cmd>()`, add a variant to `Command` in `src/cli/mod.rs` and wire its match arm, document under `docs/cli/` and add to `mkdocs.yml` nav.
- **New registry model**: add an entry to `registry.json`, verify with `fox models` and `fox pull <alias>`.

## Why these choices (load-bearing for changes)

- llama.cpp is a **submodule, not a system dep** — guarantees binding/library version parity; this is why `--recurse-submodules` is mandatory.
- `DashMap` registry = lock-free concurrent reads (looked up every request, loaded rarely).
- mpsc token channel decouples engine loop from HTTP handler and detects disconnects via `send().is_err()` without polling.
